# Architecture Decisions

이 문서는 프로젝트별 주요 기술 의사결정과 트레이드오프를 기록하는 공간입니다.

## ADR-001: Paddle Webhook을 AWS SQS로 처리

### Context

결제 Webhook은 네트워크 지연, 중복 이벤트, 외부 SaaS API 실패 가능성이 있습니다.

### Decision

Webhook 이벤트를 바로 처리하지 않고 AWS SQS에 넣은 뒤 순차적으로 꺼내 처리하는 구조를 사용했습니다.

### Why

- 이벤트 유실 가능성 감소
- 재시도 구조 구현 용이
- 외부 API 장애 시 완충 가능
- 중복 이벤트 방어 및 멱등성 설계 적용 가능

### Trade-offs

- 시스템 구성 요소 증가
- 로그와 모니터링이 더 중요해짐
- 단순한 동기 처리보다 초기 구현 비용 증가

## ADR-002: 매출 데이터 변환을 Lambda가 아닌 Glue/Athena 중심으로 처리

### Context

Paddle, PayPal, Apple IAP 등 결제 채널별 데이터 구조가 달라 신규/기존 구매자 분류 및 매출 분석이 복잡했습니다.

### Decision

원천 데이터는 S3에 저장하고, 변환/분석은 Glue/Athena에서 수행하는 구조를 사용했습니다.

### Why

- 원천 데이터 보존
- 재처리 가능성 확보
- 결제 채널 간 이력 비교 가능
- 분석 로직 변경 시 재집계 용이

## ADR-003: 레거시 플랫폼 ECS 이전은 중단하고 문서화/핸드오버 우선

### Context

EC2 기반 WordPress 레거시 플랫폼을 ECS/Fargate로 이전하면 비용 절감 가능성이 있었으나, DB 설정, 에셋 로딩, 도메인 의존성, WordPress 특성이 얽혀 있었습니다.

### Decision

남은 일정 내 무리한 마이그레이션보다 작업분을 정리하고 GitHub에 올린 뒤, 추가 작업이 가능한 상태로 문서화하는 방향을 선택했습니다.

### Why

- 제한된 일정에서 운영 장애 리스크가 큼
- 이미 30~40시간을 투자했으나 DB/에셋 이슈가 남아 있음
- 안정적인 핸드오버와 리스크 관리가 더 중요함
